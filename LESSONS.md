# HeXO Lessons Learned

- **Variable shadowing in MCTS**: Using the same variable name (`node`) for tree traversal and as a dict comprehension variable during expansion silently breaks the backup phase. Always use distinct names for tree nodes vs. board coordinates.
- **Per-graph softmax is critical**: When using PyG `Batch`, all node features are concatenated. Applying `softmax(dim=0)` across the entire batch treats nodes from different games as one distribution. Must use scatter operations to compute per-graph softmax.
- **Policy targets must be aligned with batching**: Naively `torch.cat`-ing policy targets from different games creates a flat vector that doesn't align with `Batch.from_data_list`. Storing targets as attributes on `Data` objects lets PyG handle alignment automatically.
- **Relative features improve learning**: Using `[is_me, is_opp, is_empty]` instead of `[is_p1, is_p2, is_empty]` means the network always sees the game from the current player's perspective. This halves the state space the network needs to learn.
- **Residual connections prevent over-smoothing**: Deep GCNs without skip connections cause all node embeddings to converge. Residual blocks + batch norm are essential for 6+ layer GNNs.
- **Dirichlet noise at root**: Without noise, early self-play games are too deterministic and the model converges to bad local optima. AlphaZero's noise injection is critical for exploration.
- **Temperature schedule**: Using τ=1 early in a game (proportional move sampling) and τ=0 late (greedy) balances exploration during training data generation.
- **Optimizer state matters**: Re-creating `Adam` every training call destroys accumulated momentum and adaptive learning rate state, severely hurting convergence.
- **Distance-1 halo is sufficient**: With 6 GCN layers, information propagates 6 hops already. A distance-2 halo creates unnecessarily large graphs that slow down training.
- **`__new__` in copy**: Using `HeXOGame.__new__(HeXOGame)` avoids calling `__init__` during game state cloning, which is slightly faster during the thousands of MCTS copies.
- **Platform-specific tuning matters**: M4 MacBook (MPS, 16GB) benefits from more games with fewer sims (fast single-core), while the Dell Precision (CUDA A2000, 32GB) benefits from deeper search (200 sims), larger batches (128), and mixed precision training via AMP/GradScaler to leverage tensor cores.
- **Mixed precision (AMP)**: The NVIDIA A2000 has FP16 tensor cores. Using `torch.amp.autocast` + `GradScaler` can speed up training batches with minimal accuracy loss.
- **Max move limit is essential**: With an untrained random model, games on an infinite hex grid can drag on for hundreds of moves without either player getting 6 in a row. Capping at ~100 moves (declared a draw) prevents the first self-play iteration from taking hours.
- **Hex Game Radius**: Expanding the allowable move radius (e.g., distance 8) from any piece enables freer gameplay instead of forcing clustered moves. However, `get_legal_moves()` and GNN graph generation (`get_graph()`) must use identical radii so MCTS target policies match available actions perfectly.
- **Turn-based UI Updates**: For games allowing multiple actions per turn (like Connect6/HeXO), restricting terminal board renders to `if game.placements_this_turn == 0:` provides a much cleaner user experience compared to redrawing between mid-turn piece placements.
- **Interactive PyGame GUI Implementation**: Transitioning off a CLI interface requires offloading the main blocking `input()` loop into a `pygame.event.pump()` check. Panning and camera systems for infinite hex grids require careful tracking of a global `(cx, cy)` coordinate offset to map raw screen space clicks to internal Axial coordinate indices reliably.
- **Internal Board Shape Stability**: In games allowing multiple actions per turn, newly placed pieces mid-turn should not aggressively expand the map's legal bounds (e.g., an 8-hex radius). Changing `get_legal_moves` and `get_graph` to only calculate the playable radius boundary from pieces placed *prior* to the current turn ensures the board configuration only internally expands at the clean end of a turn, stabilizing GUI drawing and MCTS targets perfectly.
- **Progress Bar TQDM Variables**: When configuring `tqdm` progress bars, ensure the `bar_format` includes the `{postfix}` variable if you intend to display dynamically updated information like move counts, win rates, or losses using `pbar.set_postfix_str()`. Without it, the data will silently fail to render in the terminal output.
- **CPU-GPU Syncs in MCTS Expansion**: GPUs handle `batch_size=1` forward passes poorly compared to CPUs. It becomes ten times worse if you index a CUDA tensor with a CPU boolean mask (e.g., `logits[cpu_is_empty]`), which forces an implicit blocking device transfer. Doing all filtering and softmax operations completely on the GPU before making a single `cpu().numpy()` call—and pairing it with `torch.inference_mode()` and `torch.amp.autocast('cuda')`—is critical to keep thousands of unbatched MCTS expansions from stalling out the device.
- **Graph Topology Caching**: When expanding allowable move radii (e.g. distance 8 from pieces), the number of nodes in the graph balloons drastically. Re-calculating node neighborhoods and converting thousands of edges to PyTorch tensors dynamically inside a Python MCTS loop (which tests thousands of node states) is cripplingly slow. Since the legal boundaries only expand after complete turns, you can heavily optimize `get_graph` by caching and cloning base structural tensors and only iterating over the handful of occupied pieces to flag internal node features!
- **Lazy MCTS State Instantiation**: If your MCTS `expand()` method aggressively creates deep copies of the game board/state for EVERY possible action (e.g., thousands of legal hex coordinates), the memory allocation and copying overhead per simulation will overwhelm your loop. Always create `MCTSNode` wrappers lazily with `state=None` and only instantiate the physical game state internally when a simulation actively traverses into that specific child node using `select_child()`.
- **GNN Graph Size Scaling**: Even with perfectly optimized Python loops, zero GPU syncs, and heavy caching, a Graph Neural Network architecture's execution time scales linearly with `O(Nodes + Edges)`. As a game like HeXO structurally unfolds across an 8-hex radial boundary, the 1-piece base matrix of 200 virtual nodes will mathematically balloon into a mesh of 4000+ nodes and 50,000+ edges by the late game. It is normal and expected for PyTorch geometric `batch_size=1` forward passes to proportionally slow down from ~1ms to ~25ms over the course of a single game due to the sheer volume of `scatter_add` convolutions occurring.
